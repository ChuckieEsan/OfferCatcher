# Memory Module Literature Review

> 主题：面向 Agent / LLM System 的记忆模块设计趋势与策略调研  
> 更新时间：2026-04-22  
> 重点范围：2023–2026，侧重 2025–2026 的最新趋势、benchmark 与工程启发

---

## 1. 这份综述回答什么问题

本文聚焦三个问题：

1. **最新的记忆模块设计趋势是什么**：研究和工业系统正在从“向量库 + top-k 检索”演进到怎样的闭环。
2. **哪些策略最值得借鉴到工程实现里**：尤其是长期对话 Agent、用户个性化 Agent、以及带工具的 Agent 系统。
3. **这些趋势对你当前的 memory 设计意味着什么**：哪些部分方向正确，哪些部分可以继续增强。

本文不是单纯罗列论文，而是按“问题—方法—趋势—工程启发”的方式组织。

---

## 2. 一句话结论

最近两年的主流趋势可以概括为：

> **Agent 记忆正在从单一的外部检索组件，演化为一个完整的 write–manage–read 闭环系统。**

也就是说，前沿系统越来越不满足于：

- 把历史对话切块
- 做 embedding
- top-k 召回
- 拼回 prompt

而是开始显式设计：

- **写入门控（what to write）**
- **记忆结构（how to structure）**
- **时间感知与重要性检索（how to retrieve）**
- **压缩与遗忘（what to forget）**
- **反思与策略更新（how memory changes behavior）**
- **评测闭环（does memory truly help the agent）**

---

## 3. 研究脉络：从“记住内容”到“经营记忆”

### 3.1 奠基阶段：让 Agent 具备持续性

这一阶段的代表工作包括：

- **MemoryBank (2023)**：强调长期对话中的用户画像和长期陪伴，提出遗忘曲线式记忆更新。
- **Generative Agents (2023)**：提出 memory stream、reflection、planning 三段式机制；用 recency、relevance、importance 联合打分做检索。
- **Reflexion (2023)**：提出把失败后的 verbal feedback 写入 episodic memory，强调“反思也是记忆”。
- **MemGPT (2023)**：借鉴操作系统的分层内存管理思想，提出不同 memory tier 间的数据移动。
- **Voyager (2023)**：用 skill library 形式保存可执行经验，表明程序性 / 技能型记忆也是长期记忆的重要形式。

这一阶段的核心贡献是把“记忆”从聊天历史扩展成一个支持长期行为一致性与能力积累的系统模块。

### 3.2 过渡阶段：从存储走向管理

到了 2024–2025，研究的重点开始从“怎么存”转向“怎么管理”：

- 不是所有内容都值得写入。
- 不是所有被写入的内容都应该长期保留。
- 检索不该只看语义相似度，而要考虑时间、重要性、用户状态、任务场景。
- 记忆要区分 **事实型（factual）** 和 **反思型（reflective / policy-like）**。

这一阶段出现了更强的 memory management 思路，例如：

- **Reflective Memory Management / RMM (ACL 2025)**
- **Mem0 / Mem0g (2025)**
- **A-MEM、AtomMem 等把 memory management 本身 agent 化或决策化的工作**

### 3.3 最近趋势：把 memory 当作长期自适应系统

2025–2026 的综述和 benchmark 普遍把现代 Agent Memory 看成一个闭环：

- **write**：从交互中筛选值得写入的信息
- **manage**：合并、反思、压缩、删除、升级、降级
- **read**：按任务目标进行条件化检索
- **act**：让记忆真正影响行为和回答
- **evaluate**：不仅评召回，还评 usefulness、efficiency、forgetting、test-time learning

这标志着记忆从“外挂存储”变成了“Agent architecture 的核心子系统”。

## 4. 最新设计趋势与策略

## 4.1 趋势一：分层记忆成为主流，而不是单一 memory store

### 现象
越来越多工作不再使用单一的“向量库承载全部记忆”的设计，而是显式区分：

- **working / short-term memory**：当前上下文、活跃状态
- **episodic memory**：过去发生过什么
- **semantic memory**：稳定知识、长期偏好、规则
- **reflective memory**：总结、策略、教训、经验
- **procedural / skill memory**：可执行技能、函数、代码、工具使用模式

### 原因
不同类型的记忆在写入频率、检索条件、生命周期和压缩方式上都不同。把它们塞进一个扁平 store 会带来：

- 冲突的写入策略
- 混杂的召回结果
- 难以控制 token 成本
- 难以建模“哪些内容会稳定影响未来行为”

### 代表工作
- MemGPT：强调不同 memory tier 的管理
- Voyager：强调 skill library 作为长期程序性记忆
- 2025–2026 的 survey：普遍反对只用“长/短期记忆”二分法，转而从 form / function / dynamics 看 memory

### 对工程的启发
如果你的系统已经有：

- `MEMORY.md`：概要 / 常驻层
- `preferences.md`、`behaviors.md`：规则 / 反思层
- `session_summaries`：情节 / 历史层

那么方向是正确的。下一步不是推翻，而是继续强化“层间迁移机制”：

- 什么样的 episodic 内容会上升为长期偏好
- 什么样的偏好应该退化、合并或删除
- 哪些内容应变成 skill / procedure，而不是普通文本记忆

---

## 4.2 趋势二：写入门控比检索优化更重要

### 现象
近年的很多工作都在减少“全量写入”的冲动，转向 **salience filtering**：

- 只写真正重要、可复用、会影响未来行为的内容
- 把一次性约束、临时上下文、情绪噪音排除在长期记忆之外

### 原因
错误写入的代价通常高于漏召回：

- 错误偏好一旦固化，会污染很多轮回答
- 过度写入会让 memory store 膨胀
- 冗余条目会降低检索质量并增加 token 成本

### 代表工作
- Mem0：强调动态提取与 consolidation salient information，而不是保存全部内容
- RMM：通过 prospective / retrospective reflection 管理 memory granularity 和 retrieval policy
- FadeMem / MemoryAgentBench：把 forgetting 与选择性保留视作核心能力，而不是副作用

### 对工程的启发
你要重点投资的是 **write-path policy**：

- 显式偏好是否立即写入？
- 多轮行为模式需要多强证据才升级为 `behaviors.md`？
- 临时约束如何只留在 working memory，而不写入长期层？

如果写入门控没做好，后面的检索、压缩、summary 质量再高都只能补救，不能根治。

---

## 4.3 趋势三：检索从纯语义相似度走向“语义 + 时间 + 重要性 + 反思”

### 现象
“embedding 相似度 + top-k”已经不够用了。最近的设计越来越重视以下联合信号：

- **relevance**：与 query 的语义相关性
- **recency**：最近是否发生或是否仍有效
- **importance**：这条记忆是否高价值
- **confidence**：这条结论是否稳定、是否多次验证
- **reflection-aware reranking**：检索结果是否应被更高层策略重排

### 原因
长期对话里最常见的问题不是“完全找不到”，而是：

- 找到了过时的信息
- 找到了相关但不重要的信息
- 找到了局部片段，却缺乏高层总结

### 代表工作
- Generative Agents：recency、relevance、importance 联合打分
- RMM：检索策略可根据 retrospective feedback 调整
- 最近的 benchmark：越来越强调 temporal relevance 和 retrieval policy，而不是单一命中率

### 对工程的启发
如果你的 `session_summaries` 检索目前主要依赖向量相似度，那么可以考虑至少加入：

- recency boost
- memory type prior（偏好 vs 历史讨论）
- conflict / supersession handling
- retrieval 后重排序

尤其在“用户偏好会变化”的场景里，时间信号往往是必需品。

---

## 4.4 趋势四：反思型记忆正在成为主线

### 现象
近年的代表性工作越来越强调：

- 记忆不只是“发生了什么”
- 更重要的是“从这件事里学到了什么”

这种记忆通常表现为：

- 经验总结
- 策略规则
- 错误归因
- 对用户习惯的抽象
- 下次应该如何做的建议

### 代表工作
- Reflexion：把 verbal critique 存入 episodic memory，用于后续 trial 改进
- Generative Agents：通过 reflection 把具体事件综合为高层抽象
- RMM：通过 prospective / retrospective reflection 在长期对话中持续形成可用的个性化 memory
- MemBench：明确区分 factual memory 与 reflective memory 两类评测对象

### 对工程的启发
你的 `preferences.md` 和 `behaviors.md` 本质上就是 reflective / policy-like memory。这个方向非常契合前沿趋势。

可以进一步增强为结构化对象，例如：

- 触发条件
- 建议行为
- 置信度
- 证据来源（哪些会话支持该模式）
- 最后验证时间

这样一来，它们不只是 markdown 文本，而是可被 agent 有条件执行的策略条目。

---

## 4.5 趋势五：选择性遗忘成为一等能力

### 现象
以往系统常把遗忘视作“容量不够时的被动裁剪”。现在越来越多论文把 forgetting 视为主动设计：

- 删除低价值记忆
- 让旧结论衰减
- 在冲突出现时用新信息 supersede 旧信息
- 保留摘要，删去细节
- 只保留 evidence links，不保留原文全文

### 原因
长期 memory store 自然会增长，而增长会带来：

- 检索延迟上升
- 噪音累积
- prompt 注入膨胀
- 旧偏好误导当前任务

### 代表工作
- MemoryBank：早期就用遗忘曲线启发更新机制
- MemoryAgentBench：把 selective forgetting 列为核心能力
- FadeMem：强调 forgetting 不是 bug，而是 memory intelligence 的组成部分

### 对工程的启发
你当前如果只有“超过 20kb 就裁掉最早注入的记忆”，这是一个工程上合理的起点，但从趋势看还偏弱。

更进一步可以考虑：

- **recency × importance** 裁剪
- **supersession**：新偏好覆盖旧偏好
- **conflict resolution**：冲突规则并存时选 active one
- **compaction**：多条相近 summary 合并成一条抽象总结

---

## 4.6 趋势六：memory management 本身开始 agent 化

### 现象
最新工作里，一个明显方向是：记忆系统不再只是固定 pipeline，而是一个“会决定如何管理记忆的 Agent / Policy”。

也就是说，系统不只是执行：

- write
- read
- update
- delete

而是会思考：

- 这条信息值不值得写？
- 该进入哪一层？
- 应该 merge 还是 overwrite？
- 现在该读 summary 还是读 rule detail？
- 哪条低价值记忆应该忘掉？

### 代表工作
- A-MEM：强调动态 memory structuring，而不是静态预定义 memory operations
- AtomMem：把 memory management 建模为 sequential decision-making
- RMM：通过 reflection 调整 memory granularity 和 retrieval policy

### 对工程的启发
你已经有 memory agent 雏形。继续往前走，可以让它不只负责：

- 写 session summary
- 改 preferences
- 改 behaviors

还可以负责：

- 合并重复记忆
- 检测并解决冲突偏好
- 决定是否从 episodic 升格到 semantic
- 给每条规则打置信度与过期时间

这会让 memory agent 从“抽取器”升级为真正的“memory manager”。

---

## 4.7 趋势七：从扁平文本记忆走向图结构和异构记忆

### 现象
越来越多工作开始尝试：

- graph-based memory
- heterogenous memory adapters
- 关系建模（用户—主题—偏好—事件）

原因是很多长期个性化信息天然是条件性的，而不是全局性的。

比如：

- 用户在 **RAG 话题** 下喜欢代码示例
- 用户在 **系统设计题** 下更喜欢表格对比
- 用户在 **基础语法类问题** 下不需要从零讲解

这类规则天然接近图结构，而不是扁平段落。

### 代表工作
- Mem0g：在 Mem0 基础上引入 graph-based memory representation
- MemAdapter：尝试在 explicit、latent、parametric 等异构 memory paradigm 间统一建模

### 对工程的启发
如果后续想增强 topic-conditional preference，这时比起继续往 markdown 里堆规则，更好的做法可能是：

- 保留 markdown 作为可读层
- 同时在后端维护结构化索引层

例如：

- topic: RAG
- preference: prefer_code_examples
- confidence: 0.92
- evidence_sessions: [s1, s7, s18]
- last_confirmed_at: ...

---

## 4.8 趋势八：评测从“能记住吗”走向“记忆是否真的帮助 Agent”

### 现象
近两年的 benchmark 越来越不满足于：

- 给一个长文本
- 问一个问题
- 看模型答对没有

而是转向测更真实的 memory competence：

- incremental multi-turn interactions
- test-time learning
- selective forgetting
- factual vs reflective memory
- observation vs participation scenario
- effectiveness、efficiency、capacity 同时考察

### 代表工作
- MemoryAgentBench：强调增量多轮交互、准确检索、长程理解、测试时学习、选择性遗忘
- MemBench：区分 factual memory / reflective memory，并同时评 effectiveness、efficiency、capacity
- 最新 survey：强调 memory evaluation 必须和实际 agent behavior 绑定，而不是只测 retrieval

### 对工程的启发
这与你的 harness 思路高度一致：

- 要评写入是否合理
- 要评检索是否正确
- 要评使用后回答是否变好
- 要评并发、遗忘、容量控制是否稳健

这基本就是当前前沿评测的正确方向。

## 5. 关键论文与其设计启发

下面按“论文—核心思想—最值得借鉴的点”整理。

### 5.1 MemoryBank (2023)
**关键词**：长期陪伴、用户画像、遗忘曲线

- 关注长期交互中的 personality understanding
- 用遗忘曲线式机制控制记忆强化与衰减
- 适合强调用户画像和陪伴式对话的场景

**启发**：
- 用户偏好不该永久等权存在
- 记忆需要 reinforcement / decay 机制

### 5.2 Generative Agents (2023)
**关键词**：memory stream、reflection、planning

- 存储自然语言 experiences
- 周期性从 experience 中产生高层 reflection
- 用 recency、relevance、importance 检索

**启发**：
- “经验 → 抽象反思”是长期记忆系统的重要升级路径
- retrieval score 应该多因子组合，而不是只看 embedding 相似度

### 5.3 Reflexion (2023)
**关键词**：verbal reinforcement、self-critique

- 将失败经验和自我反思写入 memory buffer
- 后续决策依赖这类 reflective memory

**启发**：
- 记忆不只存用户信息，也可以存系统自己的经验教训
- 如果你的 Agent 会频繁失败重试，反思记忆很有价值

### 5.4 MemGPT (2023)
**关键词**：memory tiers、virtual context management

- 把 memory 看成类似 OS 的快慢层级
- 通过跨层移动缓解上下文窗口限制

**启发**：
- memory 结构设计应该服务于 token budget
- 常驻层与外部层分离是合理的

### 5.5 Voyager (2023)
**关键词**：skill library、lifelong learning

- 用可执行技能库作为长期记忆
- 强调 skill reuse 和 compositionality

**启发**：
- 记忆不只是 facts / preferences，也可以是 executable procedure
- 你的“memory = skill”抽象在这个角度是有前沿对应物的

### 5.6 Mem0 / Mem0g (2025)
**关键词**：production-ready、salient extraction、graph memory

- 动态提取对话中的高价值信息
- 进行 consolidation 和 retrieval
- 增强版使用 graph-based memory 表示复杂关系

**启发**：
- 对生产系统而言，最关键的是“提取—整合—召回”的稳定性
- 如果 topic-conditional preference 很重要，图结构会比纯文本更有优势

### 5.7 Reflective Memory Management / RMM (ACL 2025)
**关键词**：prospective reflection、retrospective reflection

- 指出传统 memory system 的两大问题：固定粒度和固定检索机制
- 通过前瞻和回顾两种 reflection 来管理写入和检索
- 在长期个性化对话场景中取得明显提升

**启发**：
- memory management 不应是固定规则，而应持续校正
- 你现在的 `preferences.md` / `behaviors.md` 更新逻辑，未来可以引入 retrospective feedback

### 5.8 MemBench (ACL Findings 2025)
**关键词**：factual vs reflective、effectiveness / efficiency / capacity

- 把记忆评测从单一准确率扩展到多维度
- 区分事实型与反思型记忆
- 区分 participation / observation 场景

**启发**：
- 你的评测 harness 应避免只看 retrieval recall
- 需要单独评 factual memory 和 reflective memory

### 5.9 MemoryAgentBench (2025)
**关键词**：incremental multi-turn、selective forgetting、test-time learning

- 贴近真实多轮交互
- 强调测试时学习和选择性遗忘
- 把 memory competence 看成 agent 能力的一部分

**启发**：
- 你的系统测试必须包含增量交互和并发时序
- forgetting 和 update correctness 需要专门测

## 6. 最新趋势对你当前设计的具体启发

下面把研究趋势映射到你这套设计。

### 6.1 你已经踩对的方向

#### A. 分层是对的
`MEMORY.md + references + session_summaries` 非常接近当前主流的：

- 常驻概要层
- 规则 / 反思层
- 历史情节层

#### B. 规则与历史分离是对的
把 `preferences/behaviors` 与 `session_summaries` 分开，符合事实型 / 反思型 / 情节型 memory 的前沿划分。

#### C. 把记忆视作影响 Agent 行为的文档是对的
这和“memory 不只是存储，更是 policy substrate”的趋势一致。

#### D. 使用 memory agent 做后处理是对的
这与当前“memory management 本身 agent 化”的方向高度吻合。

### 6.2 你最值得补的几块

#### A. 时间感知检索
当前如果 `session_summaries` 主要靠向量召回，建议补：

- recency boost
- active / stale rule handling
- superseded preference suppression

#### B. 冲突处理
当用户偏好发生反转时，系统应：

- 显式覆盖旧偏好
- 保留版本链或 evidence
- 避免在 prompt 注入时同时出现冲突规则

#### C. 选择性遗忘
除了窗口裁剪，还应设计：

- 低价值 summary 淘汰
- 重复 summary 合并
- 长期未命中的规则降权
- topic-specific rules 的生命周期管理

#### D. 结构化 reflective memory
`behaviors.md` 和 `preferences.md` 未来可以进一步结构化，至少引入：

- scope / topic
- confidence
- evidence_count
- last_confirmed_at
- last_used_at

#### E. 检索后的使用验证
需要有指标回答：

- 取回的记忆有没有真正改善回答？
- 有没有引入错误 personalization？
- 有没有造成无关历史泄露？

## 7. 一个更前沿、但仍可落地的 memory 设计蓝图

如果把最新趋势压缩成一套适合工程实现的策略，我会建议以下蓝图。

### 7.1 四层结构

1. **Working Memory**
   - 当前对话窗口
   - 临时约束
   - 不写入长期层

2. **Episodic Memory**
   - `session_summaries`
   - 支持语义与时间检索
   - 保存关键交互片段摘要

3. **Semantic / Reflective Memory**
   - `preferences.md`
   - `behaviors.md`
   - 以及后端结构化规则表

4. **Procedural Memory**
   - skills
   - reusable prompting policies
   - reusable tools / routines

### 7.2 写入策略

- **显式偏好**：高优先级，立即候选写入
- **弱显式偏好**：需要二次确认或多次证据
- **行为模式**：需要多轮观察升级
- **任务事实**：进入 episodic，不直接写入长期规则
- **失败教训**：进入 reflective / procedural memory

### 7.3 检索策略

- 基于 query 判定优先检索哪一层
- episode 检索用 `semantic + recency + importance`
- rule 检索用 `topic match + confidence + freshness`
- skill 检索用 trigger condition + task similarity

### 7.4 遗忘策略

- summary 级别：低命中、低重要性、陈旧内容可删除
- rule 级别：有 superseding evidence 时覆盖旧规则
- skill 级别：低使用率技能可降权而不立即删掉

### 7.5 反思策略

- 周期性 consolidation
- retrospective review：哪些记忆被使用过？是否提升了回答？
- 对错误召回和错误 personalization 做回溯修正

## 8. 文献调研后的最终判断

### 8.1 当前最稳妥的主流路线
如果目标是“长期个性化对话 Agent / 工具型 Agent”，当前最稳妥的路线是：

> **分层记忆 + 强写入门控 + 时间感知检索 + 反思型规则层 + 选择性遗忘 + 面向行为改善的评测**

### 8.2 你的方案与当前趋势的匹配度
我会给出这样的判断：

- **总体方向：高匹配**
- **分层设计：匹配主流**
- **反思型规则层：非常匹配**
- **agent 化管理：匹配最新趋势**
- **时间感知检索 / 冲突处理 / 遗忘策略：仍有明显可提升空间**

### 8.3 最值得优先迭代的三个方向

1. **时间感知与冲突感知检索**
2. **显式遗忘与 consolidation 机制**
3. **把 preferences / behaviors 结构化为可执行策略对象**

## 11. 给后续设计工作的建议

如果下一步继续深化你的 memory 方案，建议直接做三件事：

1. **补一版 memory object schema**  
   给 `preferences / behaviors / summaries / skills` 定义统一元数据字段。

2. **补一版 retrieval policy spec**  
   明确 relevance、recency、importance、confidence 的组合方式。

3. **补一版 forgetting & consolidation spec**  
   明确哪些内容会 merge、overwrite、decay、delete。

这三件事做完，你的系统会从“有记忆”升级成“会经营记忆”。
