# 异步面经解析与答案生成

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                        用户上传面经                                  │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  POST /extract/submit                                                │
│  → ExtractTaskService.submit()                                       │
│  → PostgreSQL: ExtractTask (status=pending)                          │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  Extract Worker（轮询 PostgreSQL）                                   │
│  → 获取 status=pending 的任务                                        │
│  → Vision Extractor 解析                                             │
│  → PostgreSQL: ExtractTask (status=completed, result=...)            │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  POST /extract/tasks/{id}/confirm                                    │
│  → ExtractTaskService.confirm()                                      │
│  → IngestionService.ingest_interview()                               │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
        ┌─────────────────────┴─────────────────────┐
        ↓                                           ↓
┌───────────────────────┐                 ┌───────────────────────┐
│  Qdrant               │                 │  RabbitMQ             │
│  Question 入库        │                 │  MQTaskMessage        │
│  (answer=None)        │                 │  (触发答案生成)        │
└───────────────────────┘                 └───────────────────────┘
        ↓                                           ↓
        │                                  ┌───────────────────────┐
        │                                  │  Answer Worker        │
        │                                  │  → LLM 生成答案        │
        │                                  │  → Qdrant 更新 answer │
        │                                  └───────────────────────┐
        ↓                                           ↓
┌─────────────────────────────────────────────────────────────────────┐
│  Qdrant: Question (answer="生成的答案")                              │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、存储分布

| 数据 | 存储位置 | 用途 |
|------|---------|------|
| **ExtractTask** | PostgreSQL | 任务状态追踪（pending→processing→completed→confirmed） |
| **Question** | Qdrant | 面经题目（含 answer 字段） |
| **MQTaskMessage** | RabbitMQ | 答案生成任务触发 |

## 三、两个异步通道

| 通道 | 触发方式 | 处理内容 | 实现文件 |
|------|---------|---------|---------|
| **Extract Worker** | 轮询 PostgreSQL | Vision Extractor 解析面经 | `extract_worker.py` |
| **Answer Worker** | 消费 RabbitMQ | LLM 生成答案 | `answer_worker.py` |

### 设计差异分析

| Worker | 模式 | 原因 |
|--------|------|------|
| **Extract Worker** | 轮询 PostgreSQL | 任务数据已在 PostgreSQL，轮询简单可靠，无需额外发 MQ |
| **Answer Worker** | 消费 RabbitMQ | 需要高并发处理（LLM 调用阻塞型），MQ 提供吞吐量控制 |

**Extract Worker 不用 MQ 的原因**：
- ExtractTask 提交时已经写入 PostgreSQL
- PostgreSQL 是可靠存储，轮询模式不会丢失任务
- 避免引入额外的 MQ 依赖

**Answer Worker 用 MQ 的原因**：
- 需要从主系统触发（入库后发消息）
- MQ 提供更好的吞吐量和并发控制
- Answer Worker 用线程池处理阻塞型 LLM 调用

## 四、状态流转

### ExtractTask 状态

```
pending → processing → completed → confirmed
                  ↓
                failed（可重试）
```

| 状态 | 含义 | 后续操作 |
|------|------|---------|
| `pending` | 待处理 | Worker 获取任务 |
| `processing` | 正在解析 | Vision Extractor 执行 |
| `completed` | 解析完成 | 用户确认入库 |
| `confirmed` | 已入库 | 流程结束 |
| `failed` | 解析失败 | 可手动重试 |

### Question 状态（隐式）

```
answer = None → answer = "生成的答案"
```

入库时 `answer=None`，Answer Worker 消费 MQ 后填充。

### MQTaskMessage 状态（隐式）

```
sent → consumed → processed
              ↓
            retry → DLQ（超过重试阈值）
```

## 五、一致性风险点

### 问题：Qdrant + RabbitMQ 无法同一事务

```
风险点：入库 → MQ 发送
┌─────────────────────────────────────────┐
│  IngestionService.ingest_interview()    │
├─────────────────────┬───────────────────┤
│  Qdrant: Question   │  RabbitMQ: 发送   │
│  answer=None        │  可能失败！       │
└─────────────────────┴───────────────────┘
         ↓                    ↓
      成功写入              MQ 丢失
         ↓                    ↓
    Question 永久无答案
```

**根本原因**：
- Question 存储在 Qdrant（向量数据库）
- MQTaskMessage 发送到 RabbitMQ
- 无法用单一事务同时覆盖两者

**传统 Outbox Pattern 的前提**：
- 业务数据 + 消息记录 → 同一个数据库
- 利用本地事务的原子性保证一致性
- 本项目：Question 在 Qdrant，Outbox 只能在 PostgreSQL，无法共享事务

## 六、补偿机制

### 补偿脚本设计

**文件**：`scripts/resend_to_queue.py`

**扫描条件**：
- Qdrant: `Question.answer = None`
- `QuestionType ∈ {KNOWLEDGE, SCENARIO, ALGORITHM}`（需要异步答案的类型）

**操作流程**：
- 重新构建 MQTaskMessage
- 发送到 RabbitMQ
- Answer Worker 重新消费

**运行方式**：
```bash
# 预览模式（只显示不发送）
PYTHONPATH=. uv run python scripts/resend_to_queue.py --dry-run

# 实际发送
PYTHONPATH=. uv run python scripts/resend_to_queue.py
```

### 幂等性保证

Answer Worker 的幂等设计（`answer_worker.py`）：

```python
def process_answer_task(task):
    # 幂等检查：先判断答案是否已存在
    existing = question_repo.find_by_id(task.question_id)
    if existing and existing.answer:
        logger.info("Answer already exists, skipping")
        return True  # 跳过，但返回成功（消息消费成功）

    # 生成答案
    answer = agent.generate_answer(question)
    question_repo.update_answer(question_id, answer)
```

**效果**：补偿脚本重发消息时，已处理的题目不会重复生成答案。

## 十、架构反思：Qdrant vs pgvector

### 问题背景

当前架构使用 Qdrant + PostgreSQL 双存储，Question 存储在 Qdrant，ExtractTask 存储在 PostgreSQL。这导致无法用同一事务保证一致性，需要补偿脚本来兜底。

**面试常见问题**：为什么选择 Qdrant 而不是 pgvector？

### 对比分析

| 维度 | pgvector | Qdrant |
|------|----------|--------|
| **一致性** | 可与 PostgreSQL 同事务 | 跨数据库，无法同事务 |
| **运维复杂度** | 单一数据库 | 双数据库依赖 |
| **数据量适用** | 十万级以内 | 百万级以上 |
| **索引类型** | HNSW, IVFFlat | HNSW, 多种量化 |
| **过滤能力** | SQL WHERE | Payload 过滤（类似） |
| **本项目需求** | ✅ 足够 | 过度 |

### 性能数据参考

- pgvector HNSW 在 10万向量时，检索延迟约 10-50ms
- 本项目预估题目数量：万级（远低于瓶颈）
- 性能差距在百万级数据才明显

### 架构决策反思

**当时的思维误区**：

```
"向量数据库"听起来更专业 → 选择 Qdrant
        ↓
忽略了实际数据量规模
        ↓
引入双存储复杂度
        ↓
无法同一事务 → 一致性风险 → 需要补偿脚本
```

**正确的决策流程应该是**：

```
1. 数据量预估 → 万级
2. 性能需求 → pgvector 10万级足够
3. 一致性需求 → 单数据库更简单
4. 运维成本 → 单一数据库更低
   → 结论：pgvector
```

### 面试回答建议

**诚实回答**：

> 目前项目使用 Qdrant + PostgreSQL 双存储，确实存在过度设计的嫌疑。
>
> pgvector 作为 PostgreSQL 插件，可以让 Question 和 ExtractTask 在同一个数据库，避免跨数据库事务问题。当前的一致性风险（入库后 MQ 丢失）就是双存储带来的。
>
> 选择 Qdrant 的初衷是认为"向量检索性能更好"，但实际上项目数据量在万级，pgvector 完全够用。双存储带来的架构复杂度远大于性能收益。

**展示思考深度**：

> 这是个架构权衡问题，我当时的决策有改进空间。
>
> **pgvector 方案的优势**：
> - Question 和 ExtractTask 同一数据库 → 可用 Outbox Pattern 保证一致性
> - 运维简单，单一数据库依赖
> - PostgreSQL 16+ 的 pgvector 已支持 HNSW 索引，性能足够
>
> **如果重来，我会选择 pgvector**：
> - 题目入库 → PostgreSQL + Outbox → MQ 发送 → 强一致性保证
> - 向量检索 → pgvector HNSW 索引 → 性能足够
> - 单一数据库，运维简单

### 结论

**诚实承认**：当前架构确实有过度设计嫌疑，Qdrant 在项目实际数据规模下并非必要。

**面试展示**：能够反思自己的架构决策，说明权衡理由，比"盲目辩护"更能展示工程成熟度。

## 十一、架构重构方案：PostgreSQL + pgvector + Outbox + MQ

### 设计目标

1. **强一致性保证**：入库与 MQ 发送在同一事务
2. **多 Worker 并发**：MQ 提供吞吐量和负载分发
3. **事件总线架构**：支持多下游服务订阅
4. **单一数据库依赖**：运维简单，pgvector 替代 Qdrant

### 整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│  入库流程（单一 PostgreSQL 事务）                                    │
├─────────────────────┬───────────────────┬───────────────────────────┤
│  Question           │  AnswerTask       │  OutboxMessage            │
│  (业务数据)         │  (任务追踪)       │  (事件记录，用于 MQ 发送) │
└─────────────────────┴───────────────────┴───────────────────────────┘
                              ↓
                       事务提交（原子性）
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  Outbox Poller（独立进程）                                           │
│  → 扫描 OutboxMessage.status = 'pending'                            │
│  → 发送到 RabbitMQ                                                  │
│  → 更新 OutboxMessage.status = 'sent'                               │
└─────────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────────┐
│  RabbitMQ（事件总线）                                                │
│  Exchange: interview.ingested                                       │
│  Routing Key: answer.required                                       │
└─────────────────────────────────────────────────────────────────────┘
                      ↓               ↓               ↓
        ┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
        │  Answer Worker  │   │  Stats Service  │   │  Push Service   │
        │  (Queue 1)      │   │  (Queue 2)      │   │  (Queue 3)      │
        │  线程池处理     │   │  统计入库数量   │   │  推送通知       │
        └─────────────────┘   └─────────────────┘   └─────────────────┘
```

### 数据库设计

```sql
-- Question 表（含向量，使用 pgvector）
CREATE TABLE questions (
    question_id VARCHAR(32) PRIMARY KEY,
    question_text TEXT NOT NULL,
    company VARCHAR(100),
    position VARCHAR(100),
    question_type VARCHAR(20),
    answer TEXT,
    embedding VECTOR(1536),
    created_at TIMESTAMP DEFAULT NOW()
);

-- 向量索引（HNSW）
CREATE INDEX idx_questions_embedding 
ON questions USING hnsw (embedding vector_cosine_ops);

-- AnswerTask 表（任务追踪）
CREATE TABLE answer_tasks (
    task_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id VARCHAR(32) REFERENCES questions(question_id),
    status VARCHAR(20) DEFAULT 'pending',
    retry_count INT DEFAULT 0,
    error_message TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP,
    
    CONSTRAINT chk_status CHECK (status IN ('pending', 'processing', 'completed', 'failed'))
);

CREATE INDEX idx_answer_tasks_status ON answer_tasks(status, created_at);

-- Outbox 表（事件记录，用于 MQ 发送）
CREATE TABLE outbox_messages (
    message_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type VARCHAR(50),        -- 'question'
    aggregate_id VARCHAR(32),          -- question_id
    event_type VARCHAR(50),            -- 'answer_required'
    payload JSONB NOT NULL,            -- MQTaskMessage 内容
    status VARCHAR(20) DEFAULT 'pending',  -- pending/sent
    created_at TIMESTAMP DEFAULT NOW(),
    sent_at TIMESTAMP,
    
    CONSTRAINT chk_outbox_status CHECK (status IN ('pending', 'sent'))
);

CREATE INDEX idx_outbox_status_created ON outbox_messages(status, created_at);
```

### 入库流程（强一致性保证）

```python
async def ingest_interview(interview):
    """入库面经题目（单一 PostgreSQL 事务）"""
    
    async with db.transaction():
        for question in interview.questions:
            # 1. Question 入库
            db.insert("questions", {
                "question_id": question.question_id,
                "question_text": question.question_text,
                "question_type": question.question_type,
                "answer": None,
                "embedding": embedding,
            })
            
            # 2. AnswerTask 入库（需要异步答案的类型）
            if question.question_type in (KNOWLEDGE, SCENARIO, ALGORITHM):
                db.insert("answer_tasks", {
                    "question_id": question.question_id,
                    "status": "pending",
                })
                
                # 3. OutboxMessage 入库（同事务）
                db.insert("outbox_messages", {
                    "aggregate_type": "question",
                    "aggregate_id": question.question_id,
                    "event_type": "answer_required",
                    "payload": {
                        "question_id": question.question_id,
                        "question_text": question.question_text,
                        "question_type": question.question_type.value,
                        "company": question.company,
                        "position": question.position,
                    },
                    "status": "pending",
                })
    
    # 事务提交后：Question + AnswerTask + OutboxMessage 三者原子性保证
```

### Outbox Poller（MQ 发送进程）

```python
async def outbox_poller():
    """扫描 Outbox 表，发送到 MQ"""
    
    while True:
        messages = db.query(
            "SELECT * FROM outbox_messages "
            "WHERE status = 'pending' "
            "ORDER BY created_at ASC LIMIT 100"
        )
        
        for msg in messages:
            try:
                # 发送到 RabbitMQ
                await mq.publish(
                    exchange="interview.ingested",
                    routing_key=msg.event_type,
                    payload=msg.payload
                )
                
                # 更新状态
                db.update(
                    "UPDATE outbox_messages "
                    "SET status = 'sent', sent_at = NOW() "
                    "WHERE message_id = ?", msg.message_id
                )
                
            except Exception as e:
                logger.error(f"MQ send failed: {e}")
                # 不更新状态，下次轮询会重试
        
        await sleep(POLL_INTERVAL)
```

### MQ 事件总线设计

```
Exchange: interview.ingested (Topic Exchange)

┌─────────────────────────────────────────────────────────────────┐
│  Routing Key: answer.required                                   │
├─────────────────────────────────────────────────────────────────┤
│  Payload:                                                       │
│  {                                                              │
│    "question_id": "xxx",                                        │
│    "question_text": "...",                                      │
│    "question_type": "knowledge",                                │
│    "company": "字节跳动",                                        │
│    "position": "后端开发"                                        │
│  }                                                              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
        ┌─────────────────────┴─────────────────────┐
        ↓                                           ↓
┌───────────────────────┐                 ┌───────────────────────┐
│  Queue: answer_tasks  │                 │  Queue: stats_events  │
│  Binding: answer.*    │                 │  Binding: *           │
├───────────────────────┤                 ├───────────────────────┤
│  Answer Worker        │                 │  Stats Service        │
│  → 生成答案            │                 │  → 统计入库数量       │
│  → 更新 answer 字段   │                 │  → 更新仪表盘         │
└───────────────────────┘                 └───────────────────────┘
```

**扩展性说明**：
- 新增下游服务只需创建新 Queue + Binding
- 发布者无需修改代码
- 实现真正的"事件总线"解耦

### Answer Worker（多实例并发）

```python
def process_answer_task(payload):
    """处理答案生成任务（幂等）"""
    
    # 1. 幂等检查
    task = db.find_by_id("answer_tasks", payload["question_id"])
    if task and task.status == "completed":
        return True  # 已处理，跳过
    
    # 2. 更新状态为 processing（分布式锁）
    db.update(
        "UPDATE answer_tasks SET status = 'processing' "
        "WHERE question_id = ? AND status = 'pending'",
        payload["question_id"]
    )
    
    # 3. 生成答案
    answer = llm.generate_answer(payload)
    
    # 4. 更新 Question.answer + AnswerTask.status（同事务）
    with db.transaction():
        db.update("questions", 
            {"answer": answer}, 
            {"question_id": payload["question_id"]}
        )
        db.update("answer_tasks", 
            {"status": "completed", "completed_at": NOW()}, 
            {"question_id": payload["question_id"]}
        )
```

**多实例并发控制**：
- `WHERE status = 'pending'` 条件更新 → 只有一个 Worker 能抢到
- RabbitMQ prefetch 控制 → 每个 Worker 同时处理 N 个任务

### 一致性保证总结

| 环节 | 保障机制 |
|------|---------|
| **入库 → Outbox** | PostgreSQL 同事务（原子性） |
| **Outbox → MQ** | Poller 重试（失败不更新 status） |
| **MQ → Worker** | RabbitMQ 持久化 + 重试 + DLQ |
| **Worker → Question** | 幂等检查 + 状态追踪 |
| **补偿兜底** | AnswerTask 表可查询重发 |

### 运行架构

```
进程组成：
├── API Server（FastAPI）
├── Extract Worker（轮询 PostgreSQL）
├── Outbox Poller（扫描 Outbox → 发送 MQ）
├── Answer Worker × N（消费 MQ，线程池）
└── Stats Service / Push Service（可选，新增下游）
```

### 与当前架构对比

| 维度 | 当前架构 | 重构架构 |
|------|---------|---------|
| **Question 存储** | Qdrant | PostgreSQL + pgvector |
| **一致性保证** | 补偿脚本 | Outbox 同事务 |
| **MQ 角色** | 任务触发（可能丢失） | 事件总线（可靠） |
| **扩展性** | 单消费者 | 多下游服务订阅 |
| **运维依赖** | PostgreSQL + Qdrant + RabbitMQ | PostgreSQL + RabbitMQ |
| **任务追踪** | 无 | AnswerTask 表 |

### 重构收益

1. **强一致性**：入库与事件发送原子性保证
2. **运维简化**：移除 Qdrant 依赖，单一数据库
3. **架构清晰**：Outbox Pattern 标准实现
4. **扩展灵活**：新增下游服务无需改入库代码
5. **可观测性**：AnswerTask 表可追踪任务状态
