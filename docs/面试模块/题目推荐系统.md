## 一、推荐系统建模

### 1.1 问题映射

| 推荐系统概念 | OfferCatcher 对应 |
|-------------|------------------|
| **用户画像** | JD 的结构化技能要求（requiredSkills + preferredSkills + softSkills） |
| **物品** | 题库中的 Question（coreEntities、questionText、company、position、questionType、difficulty） |
| **物品特征** | 类别特征（coreEntities, questionType）、文本特征（questionText, answer）、公司/岗位标签 |
| **用户-物品交互** | 无（冷启动场景 —— 面试前不存在答题历史） |
| **推荐目标** | 最大化 JD 技能覆盖 + 题目相关性 + 类型多样性 |

### 1.2 为什么是冷启动

面试推荐天然是冷启动：用户在面试前没有与题库的交互历史。这也是设计原则之一——**面试 Agent 不应知道用户的历史掌握程度**（记忆隔离）。

这意味着不能用协同过滤，只能走**基于内容的推荐（Content-Based Recommendation）**。

### 1.3 推荐策略：Content-Based + MMR 重排

```
召回：多路召回 → 保证候选集覆盖广
排序：多维打分 → 保证每个候选有合理的相关性分数
重排：MMR 去重打散 → 保证 JD 技能全覆盖 + 题型多样性
截断：取 top N → 最终出题队列
```

## 二、整体 Pipeline

```
JD 结构化技能（requiredSkills + preferredSkills）
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│ 1. 召回（Candidate Generation）                                │
│    多路召回，每路独立产出候选题目                                │
│                                                              │
│  通道1: coreEntities 精确匹配 → PostgreSQL ILIKE              │
│  通道2: 嵌入语义召回        → Qdrant + BGE-M3 embedding      │
│  通道3: 知识图谱关联召回     → Neo4j Entity 关联              │
│  通道4: 公司+岗位泛召回     → Qdrant company+position embedding│
│                                                              │
│  每路输出：(question_id, recall_channel, raw_score)           │
└──────────────────────────────────────────────────────────────┘
    │ 候选集 C = ∪ 各通道结果，按 question_id 去重
    ▼
┌──────────────────────────────────────────────────────────────┐
│ 2. 排序（Scoring）                                            │
│    对每个候选计算多维得分                                       │
│                                                              │
│  score = α₁·语义相似度 + α₂·关键词匹配度 + α₃·来源通道权重    │
│                                                              │
│  语义相似度：Qdrant cosine score（或 reranker score）          │
│  关键词匹配度：JD skill name 与 question.coreEntities         │
│              的 Jaccard / 编辑距离                            │
│  来源通道权重：精确匹配 > 语义召回 > 图谱关联 > 泛召回          │
└──────────────────────────────────────────────────────────────┘
    │ 带分数的候选集
    ▼
┌──────────────────────────────────────────────────────────────┐
│ 3. 重排（Diversification）                                    │
│    MMR 算法 + 覆盖约束                                        │
│                                                              │
│  MMR(c) = λ·Relevance(c) - (1-λ)·max_similarity(c, selected) │
│                                                              │
│  约束1: 每个 requiredSkill 至少 1 题                           │
│  约束2: preferredSkill 尽量覆盖（鼓励但不强制）                  │
│  约束3: 题型分布均匀（knowledge 和 scenario 交替）              │
│                                                              │
│  未被覆盖的 requiredSkill → 降低召回阈值重试 → 标记缺失         │
└──────────────────────────────────────────────────────────────┘
    │ 重排后的有序队列
    ▼
┌──────────────────────────────────────────────────────────────┐
│ 4. 截断（Final Selection）                                    │
│    取 top N，按 JD skill 分组输出最终面试题目列表                │
│    输出覆盖报告：哪些 skill 覆盖了，哪些缺失                     │
└──────────────────────────────────────────────────────────────┘
```

## 三、召回层

### 3.1 通道 1：coreEntities 精确匹配

**确定性最强**。对每个 JD skill name，在 PostgreSQL 中做模糊匹配。

```sql
SELECT DISTINCT q.* FROM questions q
JOIN question_entities qe ON q.id = qe.question_id
WHERE q.user_id = :userId AND (
    qe.core_entities ILIKE CONCAT('%', :skillName, '%')
    OR q.question_text ILIKE CONCAT('%', :skillName, '%')
)
LIMIT :limit
```

**特点**：
- 精确命中，不存在语义漂移
- 依赖 `coreEntities` 标签质量——如果题库标签稀疏，命中率会低
- 按 `skill.evidence` 中的关键词做补充匹配（如 JD 写"RocketMQ 消息事务"，可拆出 "RocketMQ" 做额外 ILIKE）

### 3.2 通道 2：嵌入语义召回

对每个 JD skill，构造丰富的检索上下文，走 Qdrant 向量检索。

```
skill: { name: "分布式事务", level: "proficient", evidence: "熟悉RocketMQ消息事务" }

→ query = "考点：分布式事务, RocketMQ, 消息队列, 事务消息, 最终一致性 | 题目：分布式事务面试题"
→ embeddingAdapter.embed(query)
→ Qdrant.search(vector, userId, topK=5)
```

**特点**：
- 弥补通道 1 的标签稀疏问题
- 可能命中语义相关但实际不适用的题目（如"分布式锁"被误纳入）
- 通过 reranker 精排可以缓解

### 3.3 通道 3：知识图谱关联召回

JD skill → Neo4j `Entity` 节点 → 同实体关联的题目。

```
MATCH (e:Entity {name: "分布式事务"})
MATCH (q:Question)-[:TAGGED_WITH]->(e)
RETURN q.question_id, q.question_text
```

如果 JD skill name 与 Entity 节点名不完全匹配，先用 embedding 找到最近的 Entity 节点。

**特点**：
- 跨公司发现关联题目——RocketMQ 相关题目在不同公司下都可能被关联
- Neo4j 的图结构天然支持关联扩展（如 Entity → Cluster → 更多 Entity → 更多题目）

### 3.4 通道 4：公司+岗位泛召回

传统方案，完整覆盖兜底。

```
query = "公司：字节跳动 | 岗位：AI Agent开发 | 面试题"
→ embedding → Qdrant → top K
```

**特点**：
- 当 JD 所有技能在题库中都找不到匹配时，至少能出公司/岗位相关的题
- 也是 Channel A 的唯一召回通道

### 3.5 召回参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 每通道 top-K | 5 | 单通道单 skill 最大召回数 |
| 总候选上限 | 50 | 合并去重后的候选集上限 |
| 精确匹配阈值 | ILIKE | 不区分大小写模糊匹配 |
| 图谱扩展深度 | 1 | Neo4j 关联跳数 |

## 四、排序层

### 4.1 多维得分

每个候选题目 `c` 的最终得分：

```
score(c) = α₁·semantic(c) + α₂·keyword(c) + α₃·channel(c)
```

| 分量 | 计算方式 | 权重 | 说明 |
|------|---------|------|------|
| `semantic(c)` | Qdrant cosine score / reranker score | 0.4 | 语义相关性 |
| `keyword(c)` | Jaccard(skill_terms, c.coreEntities) | 0.35 | 关键词精确重叠度 |
| `channel(c)` | 通道权重映射 | 0.25 | 精确匹配=1.0, 语义=0.7, 图谱=0.5, 泛召回=0.3 |

### 4.2 关键词匹配度（Jaccard）

```
JD skill "分布式事务" 拆词 → {分布式, 事务, RocketMQ, 消息}
Question.coreEntities → {分布式系统, 消息队列, 事务}

Jaccard = |{分布式, 事务, 消息} ∩ {分布式系统, 消息队列, 事务}| / |{分布式, 事务, RocketMQ, 消息, 分布式系统, 消息队列}|
        = 2 / 6 = 0.33
```

## 五、重排层

### 5.1 MMR 算法

标准 MMR 公式：

```
MMR(c) = λ·Relevance(c) - (1-λ)·max_{s∈S} similarity(c, s)
```

- `Relevance(c)`：排序层算出的 score
- `max similarity(c, S)`：候选 c 与已选集合 S 中任意题目的最大相似度
- `λ = 0.7`：偏向相关性，但保留足够的多样性惩罚

### 5.2 相似度计算

两题之间的相似度：

```
sim(a, b) = β·cosine(embed(a), embed(b)) + (1-β)·Jaccard(a.coreEntities, b.coreEntities)
```

- `β = 0.6`：以语义相似为主，关键词为辅

### 5.3 覆盖约束

MMR 的贪婪选择过程中插入硬约束：

```
每次选下一题时：
  1. 检查哪些 requiredSkill 还缺题
  2. 若某 skill 缺题且候选队列中有该 skill 的候选 → 从该 skill 候选中选 MMR 最高的一题
  3. 若所有 requiredSkill 都有题了 → 正常 MMR 选取
  4. 若某 skill 的候选队列为空 → 降低召回阈值重试 → 仍无结果 → 标记 "缺失覆盖"
```

### 5.4 题型打散

```
相邻两题避免同类型：
  若 MMR 选出的下一题 type == 上一题 type → 跳过该候选，选次高的 → 连续跳过 3 次后放行
```

## 六、双渠道统一

### 6.1 入口分发

```java
// InterviewAgent.preloadQuestions()
List<InterviewQuestion> preloadQuestions(InterviewSession session) {
    if (session.getJdContext() != null && session.hasRequiredSkills()) {
        return jdDrivenSelection(session);   // Channel B: 推荐系统
    } else {
        return masteryWeightedSelection(session); // Channel A: 现有逻辑
    }
}
```

### 6.2 Channel A（无 JD）

现有逻辑保持不变：公司+岗位 embedding → mastery 60:30:10 加权 → shuffle。

可以将它视为推荐系统的**降级模式**：
- 只有通道 4（泛召回）
- 排序仅基于 mastery 权重
- 无 MMR 重排

### 6.3 Channel B（有 JD）

完整推荐 pipeline：4 通道召回 → 多维排序 → MMR 重排 → 覆盖约束截断。

| | Channel A | Channel B |
|---|---|---|
| 召回 | 仅通道 4（泛召回） | 4 通道全开 |
| 排序 | mastery 权重 | 多维得分（语义+关键词+通道） |
| 重排 | shuffle 随机 | MMR 去重打散 |
| 覆盖约束 | 无 | requiredSkill ≥ 1 题 |
| （优）简单，延迟低 | （优）精准覆盖，题型多样 |
| （劣）覆盖无保证 | （劣）延迟稍高（多路召回+计算） |

## 七、冷启动与覆盖缺口

### 7.1 覆盖缺口处理

当某个 `requiredSkill` 在所有召回通道中命中为 0：

```
1. 降低阈值重试：
   - 通道1: ILIKE → LIKE 前缀匹配
   - 通道2: topK 5 → topK 10, 去掉公司/岗位限制
   - 通道4: 放宽到全网搜索

2. 仍无结果 → 标记 "缺失覆盖"
   → 写入 session 日志
   → LLM 可考虑为该 skill 生成新题目（future work）
```

### 7.2 最坏情况兜底

所有 JD skill 在题库中都没有对应题目 → 降级到 Channel A（泛召回），保证面试仍有题可出。

## 八、实现优先级

### Phase 1：召回通道 1+2 + 基础排序（P0，3天）

1. 实现 coreEntities 精确匹配（PostgreSQL ILIKE）
2. 实现嵌入语义召回（复用现有 Qdrant search）
3. 实现基础多维打分（语义 + 关键词 + 通道权重）
4. 实现入口分发逻辑
5. 测试：有 JD 时走 Channel B，无 JD 时 Channel A 行为不变

### Phase 2：MMR 重排 + 覆盖约束（P1，2天）

1. 实现 MMR 算法
2. 实现 requiredSkill 覆盖约束
3. 实现题型打散
4. 测试：5 个 requiredSkill 每个至少 1 题，相邻两题不重类型

### Phase 3：知识图谱召回 + 覆盖缺口处理（P2，2天）

1. 实现 Neo4j Entity 关联召回
2. 实现覆盖缺口标记和降级重试
3. 实现覆盖报告输出

## 附录：与 interview-guide 的区别

| | interview-guide | OfferCatcher |
|---|---|---|
| 出题方式 | LLM 生成题目 | 题库检索推荐 |
| 题目来源 | 无题库，实时生成 | 预建题库 + Qdrant 向量索引 |
| JD 利用 | 注入 prompt 指导生成 | 作为用户画像驱动多路召回 |
| 覆盖保证 | LLM 自主决策（不可控） | MMR + 硬约束（可验证） |
| 降级策略 | 默认题库 | 泛召回通道兜底 |
